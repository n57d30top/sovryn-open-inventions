# Next Method Directions

| Direction | Why it matters | How tested | What falsifies it | Why not just framework |
| --- | --- | --- | --- | --- |
| benchmark protocol extraction and validation | The biggest repeated limitation in Batch 9 and Batch 11 was controlled split evaluation without official protocol reproduction. | Extract source protocol, implement official split, compare against random split, publish divergence. | If source protocol cannot be identified or random/official split differences are negligible across several targets. | It is target-bound: every output must run on a named external dataset. |
| split-risk detector | Vehicle fallback, Dry Bean split limits, and Optical contributor splits show split choice can dominate interpretation. | Run official, random, stratified, and group-like splits where source metadata allows. | If it only restates sklearn split parameters and finds no measurable divergence. | It is a small analysis instrument attached to executed benchmark targets. |
| leakage-risk auditor | Current tools cannot prove absence of leakage; a scoped auditor could look for target-coded columns, duplicate crossing, and group leakage. | Inject known leakage controls and detect them before applying to public targets. | If ordinary pandas checks catch every case with no added packaging or risk ranking. | It is only accepted if tested on positive and negative leakage controls. |
| repo test execution summarizer v2 | pytest_repro_summary is support-only until static inventory is reconciled with runtime collection. | Compare AST inventory, pytest collection, and executed cases across several public repos. | If dynamic tests still make counts unreliable without target-specific manual work. | It must be built only inside a repo/test reproduction batch. |
| negative-control generator | metric_stress_validator is useful because it forces controls; broader negative controls could prevent overclaiming. | Generate shuffled-label, permuted-feature, duplicate-crossing, and majority-class controls for executed targets. | If controls do not change conclusions beyond existing metric stress reports. | It is evaluated only through public target runs and baseline-dominance checks. |
| dataset-card-to-execution-plan extractor | Source metadata often names files, splits, and limits that controlled split scripts ignore. | Extract execution plans from UCI pages and verify against actual loaded files. | If manual source cards remain faster and more reliable. | It produces no standalone platform; it emits target-specific reproduction plans. |
| baseline-dominance challenge generator | Batch 5 and later batches show simple baselines are the best anti-hype guard. | Automatically choose dummy, linear, tree, and source-protocol baselines for a public target. | If selected baselines are weaker than obvious hand-picked baselines. | It is accepted only when it runs against concrete external benchmarks. |
